# 智能穿衣指南 - 数据库设计文档

> **项目**：ClosetAI / 衣橱管家
> **版本**：V1.1
> **日期**：2026-03-20
> **状态**：草稿

---

## 一、数据库架构

### 1.1 技术选型

| 类型 | 技术 | 说明 |
|------|------|------|
| 主数据库 | PostgreSQL 15+ | 存储核心业务数据 |
| 缓存 | Redis | 天气数据缓存、会话管理 |

### 1.2 架构特点

- **用户识别**：基于浏览器指纹生成唯一 userId，无需传统登录
- **标签存储**：衣物属性使用 JSONB 字段，兼顾灵活性与查询能力
- **穿搭快照**：穿搭历史采用快照模式，保留历史数据独立性
- **软删除机制**：衣物删除后保留 3 天，期满自动清理
- **数据安全**：启用 Row Level Security，确保用户数据隔离

#### 2.2.1 用户表 (users)

存储用户基本信息，通过浏览器指纹识别。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | UUID | PK, DEFAULT uuid_generate_v7() | 主键（时间有序） |
| device_fingerprint | VARCHAR(255) | UNIQUE, NOT NULL | 浏览器指纹 |
| last_active_at | TIMESTAMP | DEFAULT NOW() | 最后活跃时间 |
| created_at | TIMESTAMP | DEFAULT NOW() | 创建时间 |
| updated_at | TIMESTAMP | DEFAULT NOW() | 更新时间 |

**索引**：
- `idx_users_fingerprint` ON (device_fingerprint)
- `idx_users_last_active` ON (last_active_at)

存储用户的个性化偏好数据。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|

**JSONB 字段说明**：

#### 2.2.3 衣物表 (clothing_items)

核心资产表，存储用户衣橱中的每件衣物。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | UUID | PK | 主键 |
| user_id | UUID | FK → users.id | 关联用户 |
| category | VARCHAR(50) | NOT NULL | 一级分类 |
| sub_category | VARCHAR(100) | | 二级分类 |
| original_image_url | VARCHAR(500) | NOT NULL | 原始图片URL |
| cartoon_image_url | VARCHAR(500) | NOT NULL | 卡通图片URL |
| tags | JSONB | NOT NULL, DEFAULT '{}' | 属性标签 |
| wear_count | INTEGER | DEFAULT 0 | 穿着次数 |
| last_worn_at | DATE | | 最后穿着日期 |
| is_deleted | BOOLEAN | DEFAULT FALSE | 软删除标记 |
| deleted_at | TIMESTAMP | | 删除时间 |
| created_at | TIMESTAMP | DEFAULT NOW() | 创建时间 |
| updated_at | TIMESTAMP | DEFAULT NOW() | 更新时间 |

**category 枚举值**：
- `top` - 上装
- `bottom` - 下装
- `outerwear` - 外套
- `dress` - 连衣裙
- `shoes` - 鞋履
- `accessory` - 配饰

**sub_category 枚举值**：

| category | sub_category |
|----------|--------------|
| top | t-shirt, shirt, blouse, sweater, hoodie, vest |
| bottom | jeans, pants, shorts, skirt |
| outerwear | jacket, coat, cardigan |
| dress | dress, jumpsuit |
| shoes | sneakers, leather_shoes, boots, sandals, slippers |
| accessory | bag, hat, scarf, jewelry, belt |

**tags JSONB 结构**：

```json
{
  "colors": ["white", "blue"],
  "material": "cotton",
  "styles": ["casual", "business"],
  "occasions": ["work", "daily"],
  "seasons": ["spring", "autumn"],
  "thickness": 3
}
```

**字段说明**：
- `thickness`: 厚度等级，1-5（1最薄，5最厚）
- `colors`: 主要颜色，最多3个
- `styles`: 风格标签
- `occasions`: 适用场合
- `seasons`: 适用季节

**索引**：
- `idx_clothing_user_id` ON (user_id)
- `idx_clothing_category` ON (category)
- `idx_clothing_deleted` ON (is_deleted) WHERE is_deleted = FALSE
- `idx_clothing_tags` GIN ON (tags)

## 三、Redis 缓存设计

### 3.1 缓存策略

| Key Pattern | Value | TTL | 说明 |
|-------------|-------|-----|------|
| `weather:{city}` | 城市天气数据 | **2小时** | 避免频繁API调用 |
| `user:session:{user_id}` | 会话数据 | 7天 | 用户会话管理 |

### 3.2 天气缓存结构

```json
{
  "city": "上海",
  "temperature": 24,
  "condition": "cloudy",
  "condition_text": "多云",
  "humidity": 65,
  "wind_level": 2,
  "rain_probability": 20,
  "updated_at": "2026-03-20T10:00:00Z"
}
```

### 3.3 缓存失效策略

- **主动失效**：天气数据更新时主动删除缓存
- **TTL 兜底**：2小时自动过期，下次请求重新获取

## 四、业务规则

### 4.1 衣物软删除规则

1. 用户删除衣物时，设置 `is_deleted = TRUE`，`deleted_at = NOW()`
2. 删除后 3 天内可在回收站恢复
3. 3 天后由后台任务彻底删除（清理衣物记录及相关图片链接）

### 4.2 穿着记录规则

1. 用户采纳穿搭方案时，更新对应衣物的 `wear_count + 1`
2. 同时更新 `last_worn_at = 当前日期`

### 4.3 穿搭历史规则

1. 仅记录用户**采纳**的穿搭，不记录浏览/切换行为
2. 穿搭快照保存完整衣物数据，与原衣物解耦

## 五、安全策略

### 5.1 Row Level Security (RLS)

启用 RLS 确保用户只能访问自己的数据，防止数据泄露。

#### 5.1.1 用户上下文设置

由于项目使用浏览器指纹而非传统认证，采用应用层设置用户上下文：

```sql
-- 创建设置用户上下文的函数
CREATE OR REPLACE FUNCTION auth.set_user_id(user_uuid UUID)
RETURNS VOID AS $$
BEGIN
  PERFORM set_config('app.current_user_id', user_uuid::TEXT, false);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- 创建获取当前用户的函数
CREATE OR REPLACE FUNCTION auth.current_user_id()
RETURNS UUID AS $$
BEGIN
  RETURN NULLIF(current_setting('app.current_user_id', true), '')::UUID;
END;
$$ LANGUAGE plpgsql STABLE;
```

#### 5.1.2 用户画像表 RLS 策略

```sql
-- 启用 RLS
ALTER TABLE user_profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE user_profiles FORCE ROW LEVEL SECURITY;

-- 创建策略：用户只能访问自己的数据
CREATE POLICY user_profiles_user_policy ON user_profiles
  FOR ALL
  USING (
    user_id = (SELECT auth.current_user_id())
  )
  WITH CHECK (
    user_id = (SELECT auth.current_user_id())
  );
```

#### 5.1.3 衣物表 RLS 策略

```sql
-- 启用 RLS
ALTER TABLE clothing_items ENABLE ROW LEVEL SECURITY;
ALTER TABLE clothing_items FORCE ROW LEVEL SECURITY;

-- 创建策略：用户只能访问自己的衣物（包括软删除的）
CREATE POLICY clothing_items_user_policy ON clothing_items
  FOR ALL
  USING (
    user_id = (SELECT auth.current_user_id())
  )
  WITH CHECK (
    user_id = (SELECT auth.current_user_id())
  );
```

#### 5.1.4 穿搭历史表 RLS 策略

```sql
-- 启用 RLS
ALTER TABLE outfit_histories ENABLE ROW LEVEL SECURITY;
ALTER TABLE outfit_histories FORCE ROW LEVEL SECURITY;

-- 创建策略：用户只能访问自己的穿搭历史
CREATE POLICY outfit_histories_user_policy ON outfit_histories
  FOR ALL
  USING (
    user_id = (SELECT auth.current_user_id())
  )
  WITH CHECK (
    user_id = (SELECT auth.current_user_id())
  );
```

### 5.2 RLS 策略说明

| 表 | 策略类型 | 说明 |
|----|---------|------|
| user_profiles | ALL | 增删改查都受用户ID限制 |
| clothing_items | ALL | 增删改查都受用户ID限制（含软删除数据） |
| outfit_histories | ALL | 增删改查都受用户ID限制 |

**注意**：策略中函数包装在 `SELECT` 中，避免 per-row 重复调用导致的性能问题。

### 6.2 创建表

```sql
-- Users 表
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    device_fingerprint VARCHAR(255) UNIQUE NOT NULL,
    last_active_at TIMESTAMP DEFAULT NOW(),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_users_fingerprint ON users(device_fingerprint);
CREATE INDEX idx_users_last_active ON users(last_active_at);

-- User Profiles 表
CREATE TABLE user_profiles (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    gender VARCHAR(20),
    style_preferences JSONB DEFAULT '[]',
    season_preference JSONB DEFAULT '[]',
    default_occasion VARCHAR(50) DEFAULT 'casual',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    CONSTRAINT unique_user_profile UNIQUE (user_id)
);

CREATE INDEX idx_profiles_user_id ON user_profiles(user_id);

-- Clothing Items 表
CREATE TABLE clothing_items (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    category VARCHAR(50) NOT NULL,
    sub_category VARCHAR(100),
    original_image_url VARCHAR(500) NOT NULL,
    cartoon_image_url VARCHAR(500) NOT NULL,
    tags JSONB NOT NULL DEFAULT '{}',
    wear_count INTEGER DEFAULT 0,
    last_worn_at DATE,
    is_deleted BOOLEAN DEFAULT FALSE,
    deleted_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_clothing_user_id ON clothing_items(user_id);
CREATE INDEX idx_clothing_category ON clothing_items(category);
CREATE INDEX idx_clothing_deleted ON clothing_items(is_deleted) WHERE is_deleted = FALSE;
CREATE INDEX idx_clothing_tags ON clothing_items USING GIN(tags);

-- Outfit Histories 表
CREATE TABLE outfit_histories (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    occasion VARCHAR(50) NOT NULL,
    outfit_snapshot JSONB NOT NULL,
    weather_snapshot JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_outfit_user_id ON outfit_histories(user_id);
CREATE INDEX idx_outfit_created_at ON outfit_histories(created_at);
CREATE INDEX idx_outfit_occasion ON outfit_histories(occasion);
```

### 6.3 自动更新 updated_at 触发器

```sql
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_users_updated_at
    BEFORE UPDATE ON users
    FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

CREATE TRIGGER update_profiles_updated_at
    BEFORE UPDATE ON user_profiles
    FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

CREATE TRIGGER update_clothing_updated_at
    BEFORE UPDATE ON clothing_items
    FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

## 七、后端集成说明

### 7.1 用户上下文设置流程

每次请求时，后端需要设置用户上下文：

```typescript
// 后端中间件示例
async function setUserContext(req, res, next) {
  const userId = req.headers['x-user-id']; // 从请求头获取 userId
  if (userId) {
    await db.query('SELECT auth.set_user_id($1)', [userId]);
  }
  next();
}

// 使用：查询用户衣物时会自动过滤
const result = await db.query('SELECT * FROM clothing_items WHERE category = $1', ['top']);
// 只返回当前用户的数据
```

### 7.2 注意事项

1. **FORCE ROW LEVEL SECURITY**：即使表所有者执行查询也会受 RLS 限制
2. **上下文必须在查询前设置**：确保每次请求都正确设置用户上下文
3. **异常处理**：未设置上下文时查询返回空结果集（安全默认）
